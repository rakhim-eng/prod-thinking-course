# Инженерное мышление в продакшене

## Урок 7: Incident report который читают

После трёхчасового инцидента команда устала. Метрики нормализовались. Все хотят домой. Ты пишешь отчёт. Две страницы текста про timeline и технические детали. Никто его не читает. Через месяц та же проблема повторяется. Классика.

Хороший incident report решает две задачи. Документирует что произошло. Даёт actionable insights чтобы избежать повтора. Плохой report просто фиксирует события. Никто не меняет процессы.

Первая часть всегда summary. Одна страница максимум. Что сломалось. Сколько длилось. Сколько стоило. Кто пострадал. Чётко и кратко. Остальное в приложении.

Вторая часть what happened. Короткая timeline ключевых событий. Не каждая минута. Только поворотные моменты. Алерт пришёл в 14:20. Откат в 15:45. Root cause в 16:30.

Третья часть root cause. Не просто "bug в коде". Объясни почему баг прошёл через code review. Почему мониторинг не поймал раньше. Почему rollback занял час.

Четвёртая prevention. Конкретные действия. Не "улучшим процессы". А "добавим алерт на connection pool в Grafana к следующему вторнику". Назначь ответственных и сроки.

Пятая lessons learned. Что команда поняла. Не банальности. Конкретно "мы всегда проверяем DB connections перед релизом нагрузки".

Шестая метрики успеха. Как измерить что prevention сработала. Latency ниже 200мс при 2x нагрузке. Или zero подобных инцидентов за квартал.

Такой report читают. Действуют по нему. Проблемы не повторяются. Команда растёт.
